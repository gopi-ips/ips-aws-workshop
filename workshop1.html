01 · IAM IDENTITY CENTER
IAM Identity Center
Why AWS SSO over long-term keys?
Eliminate the risk of static credentials sitting in ~/.aws/credentials

❌ Problem: Long-term Access Keys

  • Keys stored on disk in plain text
  • Easy to accidentally commit to git
  • Hard to rotate across many developers
  • No expiry — leaked key = permanent risk
  • No MFA enforcement possible

✅ Solution: IAM Identity Center SSO

  • Short-lived credentials (1–8 hrs TTL)
  • Centralized via org access portal
  • MFA enforced at the portal level
  • One login → multiple accounts/roles
  • Audit trail via CloudTrail automatically

🏢 IPS Organization Details

SSO Start URL:
https://d-9d677c6f3a.awsapps.com/start

SSO Region: ca-central-1

Profile Name: dev

🔄 How Short-lived Creds Work

1. You run aws sso login
2. Browser opens → authenticate + MFA
3. SSO issues temporary STS token
4. Token cached locally (auto-expires)
5. CLI uses token transparently

02 · BASTION HOSTS
Traditional Architecture
🏰 What Is a Bastion Host?

📖 Definition

A bastion host (jump server) is a hardened EC2 instance in a public subnet that acts as the only entry point into private instances via SSH/RDP. Engineers SSH into bastion first, then hop to private servers.

Traditional Access Pattern
# Engineer's laptop → Bastion → Private EC2
ssh -i key.pem ec2-user@BASTION_PUBLIC_IP
# (now on bastion)
ssh -i key.pem ec2-user@10.0.1.45
# Multiple hops, multiple keys to manage

⚠️ Architecture Requirements

  • Bastion in public subnet with Elastic IP
  • Security group: port 22 open to 0.0.0.0/0 or office IPs
  • Private instances: port 22 open to bastion SG
  • SSH keys distributed to all engineers

02 · BASTION HOSTS
Why Bastions Are a Problem
❌ The Hidden Costs of Bastion Hosts

🔓 Security Risks

  • Port 22 publicly exposed
  • Brute force / CVE attack surface
  • SSH keys on engineer laptops
  • Leaked keys = permanent access
  • No session recording built-in
  • Hard to enforce MFA

🕵️ Audit Blind Spots

  • Who ran what command? Unknown.
  • SSH sessions not recorded
  • No CloudTrail for SSH activity
  • Compliance frameworks require logs

💸 Operational Overhead

  • EC2 instance cost 24/7
  • OS patching responsibility
  • Key distribution to new team members
  • Key revocation when staff leave
  • Bastion HA / failover setup

📏 Scalability Issues

  • Multiple regions = multiple bastions
  • Multi-account = complex peering
  • Can become a bottleneck

⚠️ Real-World Attack Scenarios

!

Key Leak via Git

Dev accidentally commits .pem file. Attacker gains permanent SSH access.

!

Departing Employee

Employee leaves. SSH key on their laptop still works until manually revoked.

!

CVE on Bastion

Unpatched OpenSSH vulnerability → bastion compromised → all private instances exposed.

04 · CONFIG MANAGEMENT
Concepts
📋 What Is Configuration Management?

📖 The Core Problem

Left unmanaged, servers drift. Engineer A manually installs nginx 1.18. Engineer B forgets. Engineer C installs 1.20. Now your fleet has 3 different configs. This is configuration drift.

✅ Solution: Desired State

Configuration Management tools let you declare the desired state of a server — "nginx 1.20 installed, port 80 open, firewall enabled" — and continuously enforce it, automatically correcting drift.

🔑 Key Principles

  • Idempotency — run 1x or 100x, same result
  • Declarative — describe WHAT, not HOW
  • Version-controlled — config as code in git
  • Drift correction — re-apply on schedule

Evolution of Config Management

1

Manual / Shell Scripts

Bash scripts run by hand. Not idempotent. Breaks if run twice. No drift detection.

2

Chef / Puppet (2005–2010)

First generation CM tools. Required dedicated server infrastructure. Complex to operate.

3

Ansible (2012+)

Agentless, YAML playbooks, SSH-based push model. Simpler, widely adopted.

4

SSM State Manager (Cloud-Native)

No SSH, no control node, IAM-based, integrated with AWS fleet management.

04 · ANSIBLE
Ansible Overview
⚙️ Ansible — Agentless Config Management

🔍 What is Ansible?

Ansible is an agentless, push-based configuration management and automation tool using SSH. You write Playbooks in YAML that describe desired state, and Ansible executes them on target hosts.

🏗️ Core Components

  • Control Node — machine running Ansible (your laptop/CI)
  • Inventory — list of target hosts (IP/hostname)
  • Playbook — YAML file declaring desired state
  • Task — single action (install, copy, service)
  • Module — built-in functions (yum, apt, copy, template)
  • Role — reusable group of tasks + variables
YAML — Example Ansible Playbook
---
- name: Configure web servers
  hosts: web_servers
  become: yes

  tasks:
    - name: Install nginx
      package:
        name: nginx
        state: present

    - name: Ensure nginx running
      service:
        name: nginx
        state: started
        enabled: yes

    - name: Deploy config file
      template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf
      notify: Restart nginx

  handlers:
    - name: Restart nginx
      service:
        name: nginx
        state: restarted

04 · STATE MANAGER
SSM State Manager
🔄 Continuous Compliance with State Manager

📖 What is State Manager?

State Manager uses Associations — links between an SSM Document and a set of targets — to continuously enforce desired configuration on a schedule. New instances matching the target tags automatically inherit the association.

🔄 The Lifecycle

1

Define Association

Link SSM Document to targets (tag: Env=prod)

2

Schedule Trigger

Run every 30 min, hourly, daily via cron/rate

3

Apply Config + Check Drift

Re-apply desired state, report compliance status

BASH — Create State Manager Association
# Install & keep CloudWatch Agent on all prod
aws ssm create-association \
  --name "AWS-ConfigureAWSPackage" \
  --targets '[{"Key":"tag:Env","Values":["prod"]}]' \
  --parameters \
    '{"action":["Install"],"name":["AmazonCloudWatchAgent"]}' \
  --schedule-expression "rate(30 minutes)" \
  --compliance-severity MEDIUM

# Run Ansible playbook via State Manager
aws ssm create-association \
  --name "AWS-RunAnsiblePlaybook" \
  --targets '[{"Key":"tag:Role","Values":["web"]}]' \
  --parameters \
    '{"playbookurl":["s3://mybucket/nginx-playbook.yml"]}' \
  --schedule-expression "cron(0 2 * * ? *)"

📊 Use Cases

  • 🛡️ Security agent always installed
  • 📦 Package versions enforced fleet-wide
  • 📁 Config files always up-to-date
  • 📊 Inventory collection on schedule

05 · RUN COMMAND
Run Command
▶️ Run Command Architecture & Usage
BASH — Run Command Examples
# Run shell script on all prod instances
aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --targets '[{"Key":"tag:Env","Values":["prod"]}]' \
  --parameters '{"commands":["yum update -y nginx"]}' \
  --output-s3-bucket-name my-ssm-logs \
  --max-concurrency "25%" \
  --max-errors "10%"

# Run PowerShell on Windows fleet
aws ssm send-command \
  --document-name "AWS-RunPowerShellScript" \
  --targets '[{"Key":"tag:OS","Values":["windows"]}]' \
  --parameters '{"commands":["Get-Service | Where-Object {$_.Status -eq \"Stopped\"}"]}'

# Check command status
aws ssm list-command-invocations \
  --command-id COMMAND_ID --details

🎯 Target Selection

  • Instance IDs — exact targets
  • Resource Tags — Env=prod
  • Resource Groups — AWS group
  • All Managed Instances

⚙️ Safety Controls

  • Max Concurrency — 25% at a time, prevent overload
  • Error Threshold — stop if 10% fail
  • S3 Output — full stdout/stderr logged
  • CW Logs — real-time stream

📄 Common Documents

  • AWS-RunShellScript — Linux shell
  • AWS-RunPowerShellScript — Windows PS
  • AWS-RunAnsiblePlaybook — Ansible
  • AWS-ApplyPatchBaseline — OS patches

🔄 Run Command vs State Manager

Run Command = one-time, ad-hoc, immediate
State Manager = recurring, drift-correcting, continuous
